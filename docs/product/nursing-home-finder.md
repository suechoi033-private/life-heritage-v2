> 작성: 오케스트레이터(전략·마케팅·PE 교차비평 종합) / 2026-06-06
> 과제 출처: 창업자 — "호스피스·요양원 실제 정보 + 이용 고객 리뷰 + 추천 로직, 장기요양 수급자가
> 안락하게 치료받을 곳을 찾아주고 어떻게 전달할지(페이지 위치·UX)까지 전체 화면 와이어프레임으로."

## 0. 한 줄 정체성

> **"요양시설을 점수 매겨 1등을 추천하는 곳이 아니라,
> 공단 평가등급(객관)과 먼저 다녀온 가족의 기록(공감)을 한자리에 모아,
> 가족이 후회 없이 스스로 고르도록 돕는 비교·의사결정 도구."**

이 한 문장이 데이터 소스·UX·카피·수익 구조 전부를 관통한다. North Star("후회 없이")와 정합하며,
이것이 일본 終活ねっと(매칭 단일수익 + 감수 없는 양산 → 6년 만에 소멸)와 잇다를 가르는 線이다.

## 1. TL;DR — 팀 합의 결정 6가지

| # | 결정 | 핵심 근거 |
|---|---|---|
| 1 | **알고리즘 랭킹 추천 엔진을 만들지 않는다.** 투명한 필터+정렬+명시적 비교로 간다. | 죽어가는 부모를 어디 모실지는 one-way·고신뢰 결정. 추천이 lead-gen 수수료로 움직인다는 의심 1회 = 신뢰 붕괴. 시니어 UX에서 블랙박스는 불신. |
| 2 | **데이터는 공공데이터포털 OpenAPI.** 크롤링 금지. | 장기요양기관 검색·시설별 상세·평가결과가 공식 개방 API로 합법 제공(활용신청만). |
| 3 | **v1 "리뷰" = 공개 별점이 아니라 ① 공단 객관데이터 ② 잇다 에디터 현장 르포 ③ 가족이 결정용으로 남기는 사적 방문기록.** 공개 보호자 후기는 설계해두되 모더레이션·승인 후 오픈(v1.5+). | 0개에서 시작하는 공개 후기는 가짜리뷰·명예훼손·시설 항의 지뢰밭. 빈 후기란은 신뢰를 깎는다. |
| 4 | **호스피스는 "도구"가 아니라 "이해·설명 콘텐츠"로 포함.** 비교·추천·필터에서는 제외, 자동 추천 안 함. | 호스피스(임종 완화의료)는 "고르는" 결정이 아니라 "받아들이는" 시점. 요양원과 한 톤으로 섞으면 존엄이 무너진다. |
| 5 | **lead-gen 상담 연결은 v1 = 신청 의사·동의 수집까지만.** 실제 외부 발송은 창업자 승인 후. | 외부 발송은 one-way door. 정보가 먼저, 매칭은 사용자가 막혔을 때 pull형으로. |
| 6 | **진입 위치 = 정보 허브 `info.html` "곁에" 탭 아래.** 신규 IA 안 만든다. | `nursing-home` 카테고리가 이미 정의돼 있고(미오픈), 매핑 2줄이면 전용 페이지로 라우팅. |

## 2. 교차비평 — 해소한 충돌 4개

세 에이전트가 독립적으로 같은 포지셔닝에 수렴했으나, 다음 4개 지점은 오케스트레이터가 조정했다.

### 충돌 A — 호스피스 스코프 (전략 "보류" vs 마케팅·PE "포함")
- 전략: v1 제외. 마케팅: N4 설명 콘텐츠 + 화면 D로 포함. PE: 유형 필터에 호스피스 옵션.
- **해소**: 창업자가 명시적으로 호스피스를 먼저 말했으므로 드롭하지 않는다. 단 **"비교·추천 도구"에서는 빼고, "이해·설명 콘텐츠 페이지(N4)"로만 v1 포함**. 전략의 "별도 콘텐츠로 분리" = 마케팅의 N4. 도구의 유형 필터에서 호스피스는 v1 비활성(준비 중 라벨).

### 충돌 B — 공개 후기 v1 포함 여부 (전략 "제외" vs PE 리뷰 작성 화면 D)
- **해소**: 둘 다 맞다. PE의 리뷰 화면·`facility_reviews` 테이블은 `status='pending'` 모더레이션 게이트 + 구현 로드맵 4단계(승인 필요)로 이미 봉인돼 있다. **v1 빌드 = 공단데이터 + 에디터 르포 + 가족 사적 비교/방문기록. 공개 보호자 후기는 설계만 하고 모더레이션·창업자 승인 후 오픈.** 화면 D는 v1.5 산출물로 명시.

### 충돌 C — "가족추천 %" 지표가 사실상 공개 랭킹 아닌가
- PE·마케팅이 별점 대체로 "가족추천 84%"를 카드에 제안. 그러나 실명 시설에 붙는 공개 추천%는 별점과 동일한 명예훼손·항의 리스크를 진다(전략 Q3 경고).
- **해소**: **v1 시설 카드·상세에는 공단 평가등급(공식·객관)만 표시.** "가족추천 %"는 공개 후기가 열리는 v1.5+에서, 후기 N건(예: ≥5) 이상일 때만 노출. v1엔 추천% 없음 → 법적으로 깨끗.

### 충돌 D — BM "매칭 30~50만" vs 본 설계 "v1 매칭 제외"
- **해소**: 매칭을 죽이는 게 아니라 **순서**다. 신뢰받는 비교 도구를 먼저 깔고 → 그 위에 상담 연결을 얹는다. 랭킹과 수수료를 구조적으로 분리(돈 낸 시설 상단 고정 금지)하면 신뢰와 수익이 충돌하지 않는다. `docs/strategy.md` §2 lead-gen 라인에 "요양 추천 = 비교도구로 정의, 매칭은 도구 뒤 단계" 결정 반영 제안.

## 3. 스코프 (v1 / v1.5 / v2)

```
v1   (도구·읽기 중심, 공개 리스크 최소)
  ├─ 공단 OpenAPI 시설 데이터(평가등급·정원/현원·인력·급여종류)
  ├─ 잇다 에디터 현장 르포 (서울/경기 핵심 30~50곳, seed-06 9기준 템플릿)
  ├─ 시설 찾기/필터/정렬 + 시설 상세
  ├─ 가족 비교표 + 사적 방문기록(체크리스트, 형제 공유)
  ├─ 호스피스 = "이해·설명" 콘텐츠 1페이지(N4)만 (도구 제외)
  └─ lead-gen = 상담 신청 의사·동의 수집 폼까지 (외부 발송 X)

v1.5 (승인 후)
  ├─ 공개 보호자 후기(5축 구조화 + 모더레이션 게이트)
  ├─ "가족추천 %" (후기 ≥5건 시)
  └─ 실제 상담 연결(외부 발송, Edge Function)

v2
  ├─ 요양원·간병 매칭 파트너 계약
  ├─ 케어링(nest) 통합 — 비교 후보 → 노트 "의료 의지" 저장
  └─ (검토) 명시적 "맞춤 정렬" 토글 — 가중치 근거 펼침형
```

## 4. IA · 페이지 위치 · 라우팅

### 진입
정보 허브(`info.html`) → "곁에" 탭 → 좌측 카테고리 "요양원·요양병원" → 전용 페이지.

### info.html 수정 (프런트만, DB 영향 0 — 정확히 2곳)
```js
// line 349 — ENABLED_KEYS에 키 추가
const ENABLED_KEYS = new Set([
  'advance-directive', 'funeral-prepay', 'long-term-care', 'nursing-home'
]);

// line 381~385 — ENABLED 라우팅 맵에 추가
'nursing-home': './info/nursing-home.html',
```
이 두 줄로 사이드바 "요양원·요양병원"이 "정리 중" empty state → 전용 페이지 직행(line 386 분기가 이미 처리). 그 외 로직 손대지 않음.

### 신규 파일 (모두 `/info/` 아래 — 기존 전용페이지 관례. 빌드 단계 없는 정적 멀티페이지)
```
info/nursing-home.html          (A) 찾기/리스트       ← 허브 도착지
info/nursing-home-detail.html   (B) 시설 상세         ?id=<facility_id>
info/nursing-home-compare.html  (C) 비교             ?ids=a,b,c
info/nursing-home-review.html   (D) 후기 작성  [v1.5] ?id=<facility_id>
info/hospice.html               (N4) 호스피스 이해·설명 콘텐츠
```
상세/비교/후기는 쿼리스트링 상태 전달. SPA 안 함 — 멀티페이지가 패턴 일관성·SW 캐싱·a11y(페이지=문서) 모두 유리.

### care(케어링/nest)와의 관계
- info의 nursing-home = **공개 정보·탐색**(로그인 불필요, SEO 유입).
- nest = **개인 돌봄 기록**. 연결은 단방향 딥링크(상세 → 노트 "의료 의지" 저장)로 v2.

## 6. 데이터 모델 초안 (제안 — 스키마 변경은 창업자 승인 필요)

> publishable 키는 RLS 보호. 실제 마이그레이션은 백업 확인 + 승인 후. 공공데이터 적재는 별도 ETL(service_role)만.

```sql
-- (a) 공공데이터 미러 — 읽기전용·신뢰원천. 공개 SELECT, 쓰기는 service_role만.
create table facilities (
  id              uuid primary key default gen_random_uuid(),
  ltc_admin_code  text unique,        -- 공단 장기요양기관기호 (멱등 upsert 키)
  name            text not null,
  type            text not null,      -- 'nursing_home'|'ltc_hospital'|'day_night'|'short_stay'
  sido text, sigungu text,            -- 지역 필터
  lat double precision, lng double precision,  -- 거리(클라이언트 계산)
  eval_grade      text,               -- 'A'..'E' (공단 평가)
  eval_year       int,
  capacity int, current_count int,    -- 정원/현원 → 여유
  staff_ratio     numeric,            -- 요양보호사 1인당
  service_types   text[],             -- 급여종류
  dementia_unit   boolean default false,
  public_updated_at date,             -- 공단데이터 기준일(신선도 표시)
  created_at timestamptz default now()
);
-- 인덱스: (sido,sigungu),(type),(eval_grade), 위치는 위경도 박스쿼리

-- (b) 1차 후기 — [v1.5] 사용자 생성·RLS 보호·모더레이션 게이트
create table facility_reviews (
  id uuid primary key default gen_random_uuid(),
  facility_id uuid references facilities(id) on delete cascade,
  user_id uuid references auth.users(id),
  relation text,                      -- 'child'|'spouse'|'self'|'sibling'
  usage_period text,
  score_care int, score_hygiene int, score_comm int,
  score_meal int, score_medical int,  -- 1~5 다축
  recommend text,                     -- 'yes'|'neutral'|'no' → 추천% 집계
  body text,
  status text default 'pending',      -- 'pending'|'published'|'hidden'
  created_at timestamptz default now()
);
```

**RLS · 데이터 안전**
- `facilities`: SELECT 공개(anon), INSERT/UPDATE RLS 차단 → service_role ETL만. 사용자가 신뢰원천을 못 바꾼다.
- `facility_reviews`: SELECT는 `status='published'`만 공개(+본인 글 본인 조회). INSERT는 `auth.uid()=user_id` 강제 + `status`를 `pending` 외로 못 넣게. 게시 승격은 관리자만. UPDATE/DELETE 본인 한정, `status` 컬럼 본인 변경 불가.
- 집계는 **뷰**(`v_facility_stats`: 후기수·추천율·축별 평균)로 — 클라이언트가 평균 계산 안 함.
- 사용자 위치는 **저장 안 함**(클라이언트 계산, 프라이버시). 상담 연락처는 별도 `consult_requests`, RLS 본인만, 외부 전달은 명시 동의 + Edge Function 로깅.

## 7. "추천 로직" 입장 (재확인)

가중치 알고리즘 자동 추천을 v1 기본값으로 두지 않는다. 이유:
1. **블랙박스 = 불신.** "왜 이 시설이 1위?"에 못 답하면 부모를 맡기는 결정에서 신뢰가 무너진다.
2. **YMYL·법적 리스크.** "추천 1위" 라벨은 시설 간 우열·광고성 시비·제휴 의혹을 부른다.
3. **구현 ROI.** 얇은 후기 풀에서 점수화는 노이즈. 정렬 select + 비교표가 ROI 압도적.

**잇다식 "추천" = 좁혀드리기**: 입력(등급·거리·예산·치매여부) → 조건 만족 리스트 + "왜 이 조건인지" 한 줄 설명(seed-06 §0 우선순위 인용). 점수 합산 랭킹 없음. v2에서 선택적 "맞춤 정렬"을 **명시적 토글 + 가중치 근거 펼침**으로만 검토.
