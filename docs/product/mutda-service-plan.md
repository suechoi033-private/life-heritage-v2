## 1. 배경과 문제 정의

**1-1. 죽음 경험의 실종**

1970년대 이후 한국에서 집에서 가족의 임종을 지키는 경험은 급격히 사라졌다.
지금 대부분의 죽음은 병원·요양원·호스피스에서 일어난다(미션 문서 및
`info/home-death-korea.html` 콘텐츠 참조 — 한국은 병원 사망이 압도적이고,
자택 사망은 대개 경찰 검안 경로로 흐른다). 그 결과 지금의 30~60대는
**죽음을 곁에서 배운 적이 없는 첫 세대**다. 유언·유품·연명치료·장례에 대해
무엇을, 언제, 어떻게 준비해야 하는지 가르쳐주는 곳이 없다.

**1-2. 준비 부재가 만드는 비용**

- **남는 가족의 부담**: 유언이 없으면 상속은 갈등이 되고, 정리되지 않은 유품·계정·
  반려동물은 슬픔 위에 실무 혼란을 얹는다. "미리 알았으면"이라는 후회가 공통 증상이다
  (사업계획서 v3 §2).
- **사회적 비용**: 국내 65세 이상 치매 환자 보유 자산은 약 154조 원으로 추산되며,
  미리 정리·위임되지 않으면 경제적 학대로 이어지기도 한다
  [출처: 보건복지부 '치매 안심 재산관리' 시범사업 관련 보도 — 사업계획서 v3 인용].
- **고독사**: 1인 가구 고령층이 늘며 "아무도 모르는 죽음"이 사회 문제가 됐다.
  혼자 사는 사람에게는 준비 이전에 **"내가 무사하다는 신호"** 자체가 필요하다.
  (규모 수치는 본 문서에서 단정하지 않는다 — 통계 인용 시 출처 확인 후 추가.)

**1-3. 시장의 빈자리 (벤치마킹 요약)**

한국 경쟁 서비스(고이·첫장·에덴에이드)는 전부 '사후' 매칭이다.
'사전 준비' 포지션은 비어 있다(사업계획서 v3 §7). 일본 슈카츠넷은 사전준비
카테고리를 열었으나 YMYL 무감수 양산 + 매칭 단일 수익으로 6년 만에 소멸했다 —
묻다가 반드시 피할 함정이다.

**한 줄 문제 정의**: 죽음을 배운 적 없는 세대가, 준비를 시작할 조용하고 믿을 수 있는
자리가 없다. 그래서 준비하지 않고, 그 비용을 남는 가족과 사회가 치른다.

## 2. 미션과 비전

**미션** (goal-2026q3.md 그대로 계승):

> 삶을 소중히 여기는 사람들이 **미리 죽음을 대비**해 나와 사랑하는 사람들의
> 고충을 덜게 한다. 고령·노령사로 인한 사회적 비용을 줄이는 데 기여한다.
> 1970년대 이후 집에서 가족의 임종을 겪는 경험이 사라진 세대가(대부분
> 요양원·호스피스에서 사망) 유품정리·유언·연명치료에 대한 생각을 미리
> 정리하도록 돕는다. 최대한 심플하게, 알아야 할 과정·절차·정보를 생각해보게 한다.

**비전**: 회사 North Star("사람들이 후회 없이 살도록 돕는다")의 '자기 준비' 관문을
묻다가 맡는다. 잘 죽는 준비는 잘 사는 방법이다 — 묻다는 죽음 앱이 아니라
**후회를 줄이는 앱**이다.

**이름의 뜻**: **묻다** — ①묻다(問): 모든 핵심 기능이 질문으로 시작한다(유언장도,
편지도, 온보딩도). ②묻다(埋): 마음과 물건의 자리를 정해 고이 묻어(남겨)둔다.

## 3. 타겟과 포지셔닝 — 투 브랜드 구조 (창업자 결정, 2026-07-06)

| | **묻다** | **잇다** |
|---|---|---|
| 역할 | **미리 준비** | **돌봄과 그 이후** |
| 범위 | 유언장 · 감사/작별인사 · 유품정리 · 반려동물 · 안부확인 | 케어링 · 요양병원/호스피스 추천 · 방문진료 · 완화의료 정보 → 향후 디지털납골당 |
| 주 사용자 | 자기 준비를 시작하려는 사람 (혼자 사는 이들 포함) | 부모를 돌보는 가족, 사별 이후의 가족 |

- **같은 계정**: 두 서비스는 하나의 Supabase Auth를 공유한다. 가입 한 번으로
  자유롭게 넘나든다.
- **상호 브리지**: 잇다 info.html ↔ 묻다 my.html·library.html에 상호 카드 상시 노출
  (2026-07-06 구현 완료).
- **lifecycle 논리**: 잇다에서 부모를 보낸 사람이 6~12개월 후 "내 차례" 준비로
  넘어오는 자리가 묻다다(전략 문서의 2→3 전환). 반대로 묻다 사용자가 돌봄 상황을
  만나면 잇다로 넘어간다. **투 브랜드는 분리가 아니라 한 사람의 시간축 분담이다.**

## 4. 핵심 가치 제안

1. **"묻고 답하면 만들어진다."**
   "유언장을 쓰세요"라고 시키지 않는다. 질문에 하나씩 답하면 유언장 초안이,
   편지가, 인계서가 만들어져 있다. 빈 화면의 공포를 구조적으로 제거한다
   (편지의 "첫 문장 받기", 온보딩의 첫 질문 quick win도 같은 원리).
2. **조용함 — 경쟁·과시 없음.**
   좋아요 수·순위·비교 없음(헌장 직접 규정). 스트릭은 "이어온 날들",
   진행 표시는 "매듭" — 남과 겨루지 않는 나만의 기록. Epic(어린이 독서앱)의
   검증된 리텐션 구조를 차용하되, 경쟁 장치는 의도적으로 걷어냈다.
3. **정직한 고지.**
   - YMYL: 유언장 등 법률 화면에 "법률 자문 아님 · 전문가 감수 진행 중 베타" 고지.
   - 웹 한계: 안부확인은 "묻다 접속" 기준임을 화면에 명시(웹앱은 폰 사용시간·
     백그라운드 GPS를 측정할 수 없다). 공증도 "대면만 가능"이라는 사실을 그대로
     알린다. **할 수 없는 것을 할 수 있는 척하지 않는 것**이 신뢰 자산이다.

## 5. 기능 명세 (현행 — 2026-07-05 배포 기준)

코드 `/mutda/` (정적 HTML+JS, 잇다 디자인 토큰 상속) · DB는 잇다와 같은 Supabase
프로젝트에 `mutda_` 접두사 테이블 11개(RLS, 소유자 전용)로 격리.
라이브: `https://suechoi033-private.github.io/life-heritage-v2/mutda/`

### 5-1. 온보딩 퀴즈 → 개인화 여정
- 가입(이메일, 3분 약속) 후 짧은 퀴즈: 연령대·방문 동기·가장 마음 쓰이는 것·
  반려동물 유무.
- 답에 따라 **개인화된 준비 여정**을 생성 — 고른 순서대로 매듭(7개)이 앞에 배치되고,
  '혼자의 안전' 동기면 안부확인이 맨 앞에 온다.
- 마지막 스텝 = 첫 성공 경험: "지금 가장 고마운 사람은?" 한 문장 →
  감사편지 초안으로 자동 저장.
- 홈은 "오늘의 한 걸음"으로 다음 매듭 1개만 제시. 스트릭 칩 "이어온 날들"(비교 없음).

### 5-2. 유언장 위저드 (`will.html`)
- 7개 질문(성명→생년월일→주소→재산과 수령인(반복입력)→유언집행자→장례 희망→
  남기는 말)에 답하면 **자필증서 유언 초안** 자동 생성. 답은 `mutda_will_answers`에
  문항별 저장 — 중단 후 이어쓰기, 수정 이력이 기록으로 남는다(유언장 버전).
- **입력 보조**: 주소는 카카오 우편번호 검색 임베드(실패 시 직접 입력 폴백).
  재산은 종류 칩(모든 재산/부동산/예금/보험/차/귀중품/디지털/기타) —
  '모든 재산'은 단독이면 포괄 문구, 병행이면 잔여 조항으로 생성.
- **상속 순위 마법사**: 배우자→자녀→부모 3답으로 민법 §1000·§1003 법정 1순위 안내
  + 유류분 고지 + 실명 입력 → 실시간 문구 미리보기 → '누구에게' 자동 채움.
- **남기는 말**: 받는 사람 칩(가족 모두/배우자/엄마/아빠/자녀/형제자매/기타) —
  사람별 칸과 맞춤 예문, 초안에 사람별로 정리.
- 초안 이후: ① 자필 옮겨쓰기 체크리스트(민법 §1066의 5요건: 전문 자서·연월일·주소·
  성명·날인) ② 자필 완료 마킹 ③ 공증 연결 — 대한공증인협회 안내 + 준비물.
  상태 머신 `draft → handwritten → notarized`.
- **법률 감수**: **법무법인 율사서재 김한나 대표변호사**(02-523-1579 ·
  hss1@yulsasj.com)가 감수 파트너로 확정(2026-07-05). 감수자 표기 + will.html에
  상담 연락처 노출. 감수 완료 전까지 "감수 진행 중 베타" 라벨 유지, 정식 발행
  (마케팅 노출) 전 감수 필수 — 헌장 규칙.
- **사실 고지**: 유언 공증(공정증서)은 대면만 가능(법무부 전자공증은 사서증서 인증만
  지원, 유언 제외). 공증은 임명 공증인 또는 공증인가 법무법인만 가능. 앱에 그대로 고지.

### 5-3. 편지 — 감사의 말·작별인사 + 리텐션 루프 (`letters.html`)
- 두 종류(감사/작별). 빈 화면 공포를 없애는 **"첫 문장 받기"**.
  철학: "보내지 않아도 됩니다. 쓰는 것만으로 시작입니다."
- **오늘의 마음 질문**: 14개 질문이 날짜별로 순환 — 매일 다시 올 이유.
  질문에서 편지쓰기로 직결.
- **전하기**: 카톡/문자 공유(navigator.share, 폴백 클립보드) → sent_at 기록 →
  "🌿 미리 전한 마음, N번째예요" 리워드 카피.
- **마음의 잔디**: 최근 12주 그리드(쓴 날 연둣빛·전한 날 진초록) —
  비교 없는 나만의 잔디.
- **익명 나눔**: 편지를 익명으로 커뮤니티(함께)에 공유/내리기 가능.
- **전체 집계**: 랜딩·편지 화면에 "오늘까지 N번의 마음이 미리 전해졌습니다" —
  내용 미노출, 숫자만 집계하는 RPC. 이 누적치가 **Q3 사회적 가치 북극성 지표**다.

### 5-4. 유품 미리 정리 (`belongings.html`)
- 품목 하나 = {이름, 분류(귀중품/추억/일상/디지털), 자리, 받을 사람, 이야기 메모}.
  "하루 하나" 리듬. 실제 정리 완료 체크 별도.
- 자리 선택지: 물려주기/기부/간직/비우기/고민 중 + **맡겨두기(돌봄 부탁)** —
  임시 보관인과 최종 수령인을 지정하고 부탁을 전한다.
- **판매하기 + 공개 동의**: 판매 의사가 있는 품목은 공개 동의를 받아 표시 —
  향후 수익모델 가설의 데이터 씨앗(§8). 작성 단계에서 익명 나눔도 가능.

### 5-5. 반려동물 돌봄 플랜 (`pet.html`)
- 비상시 인계서: 식사·건강·주치의·비상 돌봄인(사전 동의 체크)·인계 메모. 인쇄 지원.

### 5-6. 안부확인 — 고독사 방지 (`checkin.html` + `guardian.html`, v2 앱푸시)
- **하트비트**: 앱을 열면 RPC가 `last_active_at` 갱신(5분 스로틀) + 미해결 알림 자동
  해제. 위치 공유 동의 시 접속 순간의 GPS 기록.
- **보호자(비상연락망)**: 이름·관계·**전화번호(필수)**·**알림 순위(1~3)** 등록 →

  (온보딩 생략, 경량 프로필) → 연결 → 웹푸시 구독. 상태 칩 "수락 대기"/"✓ 연결됨".
  **보호자 초대 = 바이럴 루프이기도 하다**(Q3 목표 지표).
- **스캔·발송**: pg_cron 30분 주기 스캔 — 임계시간(기본 18h, 12/18/24/48 선택) 초과 시

  ②**지정순위 캐스케이드 웹푸시**(1순위 실패 시 다음 순위). 이메일 발송은 폐기

## 6. 유저 저니

```
[가입]        이메일 3분 가입 (보호자 초대 링크로 온 사람은 경량 가입)
   ↓
[온보딩]      퀴즈 4문항 → 개인화 여정(매듭 7개) 생성
   ↓          → 첫 질문 "지금 가장 고마운 사람은?" 한 문장 = 첫 성공 경험
[첫 기록]     그 답이 감사편지 초안으로 저장돼 있음 — "벌써 시작했네"
   ↓
[일일 루프]   홈 인사 + 이어온 날들 + "오늘의 한 걸음" 1개
              + 오늘의 마음 질문(14개 순환) + (켠 사람은) 안부 하트비트
   ↓
[주간 리듬]   여정 매듭 하나 완성 → 유언장 버전·편지 수·품목 수가 자산으로 쌓임
   ↓
[나눔]        편지 전하기(카톡/문자) · 익명 나눔 · 보호자 초대
              → "미리 전한 마음 N번째" 집계 = 개인 행동이 집단의 아름다움으로
   ↓
[브리지]      돌봄·사별 상황을 만나면 → 잇다(같은 계정, 카드 한 번 탭)
              부모를 보낸 잇다 사용자가 "내 차례" 준비로 → 묻다
```

핵심 설계: 리텐션의 두 기둥은 **①매일의 질문(마음)과 ②안부확인(안전)**이다.
죽음준비는 데일리 카테고리가 아니지만, 이 둘이 매일 열 이유를 만든다는 것이
이번 분기의 핵심 검증 대상이다(§7, §10).

## 10. 리스크와 원칙

**10-1. YMYL 감수 정책**
- 유언장·법률 콘텐츠는 김한나 변호사 감수 완료 전까지 "베타" 라벨 유지,
  정식 발행(마케팅 노출) 금지. 의료 정보(Phase 3)는 의료 감수 확보 전 미착수.
  감수자 명단은 화면에 표시(visible) — 슈카츠넷 실패의 반면교사.

**10-2. 개인정보 — 민감정보 취급**
- 묻다가 다루는 데이터는 유언 내용·재산·가족관계·건강(반려동물 포함 인계 정보)·
  위치(GPS)·보호자 전화번호 등 최고 민감도다. 현행: 전 테이블 RLS 소유자 전용
  (커뮤니티만 읽기 공개), VAPID 개인키 Vault 보관, 위치는 **동의 시에만** 기록,
  편지 집계는 내용 미노출 숫자만.
- 과제: 개인정보처리방침 공통 템플릿(창업자 원칙 #3) 제작이 선행 과제로 남아 있음.
  정식 런칭 전 필수. 베타용 가입 즉시 확인 트리거는 커스텀 SMTP 연결 후 제거 권장
  (worklog 2026-07-05).

**10-3. 제품 리스크 (정직한 자기 평가)**
- **리텐션 가설 미검증**: "질문 순환 + 안부확인이 데일리 방문을 만든다"는 아직
  가설이다. D7 25%가 안 나오면 루프 재설계가 우선이지 채널 확장이 아니다.
- **보호자 초대 마찰**: 잇다 형제 초대에서 배운 것 — 진짜 장벽은 UI가 아니라
  관계적 마찰일 수 있다. 미초대자 정성 확인 필수.
- **웹 한계**: 접속 기준 안부확인은 오탐(폰은 쓰는데 묻다만 안 연 경우)이 구조적으로
  존재. 화면 고지로 관리하되, 근본 해결은 Phase 2.

**10-4. 헌장 준수**
- 조용함·존엄. 과시·경쟁 금지 — 좋아요 수·순위·비교 없음(스트릭·잔디도 나만 본다).
- 추모(memorial)는 보류·숨김 — 디지털납골당은 창업자 승인 후에만.
- 되돌리기 어려운 행동(돈·계약·외부 발송·DB 파괴)은 창업자 승인.

**10-5. 창업자 4원칙 (`docs/company/app-principles.md`)**
1. 배포 스크립트 공통 재사용(`.github/workflows/pages.yml` — 묻다도 이 파이프라인)
2. 새 화면은 기존 디자인 시스템·토큰부터(묻다 CSS는 잇다 토큰 상속, 임의 hex 금지)
3. 개인정보처리방침 공통 템플릿(제작 선행 과제)
4. 앱 간 중복 코드 금지(잇다 auth.js ↔ 묻다 app.js 유사 구조 — 다음 신규 앱 전
   `shared/` 추출 검토, Flutter 전환 시 packages로 구조 해결)
