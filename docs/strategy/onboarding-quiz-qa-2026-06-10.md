## 0. 사실관계 확인 (코드에서 직접 확인)

파일: `onboarding.html` L141–195 (STEPS), L199–213 (computeStage), L216–249 (결과 카피).
정합성: 5/26 확정 문서와 카피 100% 일치, 분기 트리는 **없음**(모두에게 동일 7문항).

### 0.1 7문항 전체 표

| # | key | type | 질문 | 옵션 (v: label) | 결과에 미치는 영향 |
|---|---|---|---|---|---|

| 2 | q2 | single | 그 마음이 든 계기가 있다면, 어느 쪽에 가까운가요? | care(부모·가족 편찮음) / loss(보내드림) / health(검진·진단) / milestone(은퇴·환갑·손주) / none(특별 계기 없음) | **stage 최종 판정의 핵심.** Q2 우선 |
| — | — | interstitial | "그러시군요. {stage 한 줄}. 몇 가지만 더 여쭤볼게요." | — | 미러링만, 분기 없음 |
| 3 | q3 | single | 이 고민을 누구와 함께하고 계세요? | siblings / spouse / children / alone | `alone`이면 `solo=true` 플래그 → 결과에 SOLO 블록 덧붙임 |
| 4 | q4 | single | 지금 마음은 어느 쪽에 가까우세요? | heavy / mid / light | **현재 결과 화면에서 사용처 없음.** 저장만 됨 |
| 5 | q5 | multi | 요즘 가장 많이 생각하는 것은? | prep / words / care / grief / record | **현재 결과 화면에서 사용처 없음.** 저장만 됨 |
| 6 | q6 | single | 지금 시작한다면, 어느 정도가 편하시겠어요? | read / answer / do | **현재 결과 화면에서 사용처 없음.** 저장만 됨 |
| 7 | q7 | text | 지금 가장 남기고 싶은 한 문장 | (자유서술, 선택) | 결과 상단 인용블록으로 그대로 노출 |

### 0.2 stage 판정 로직 (L199–213)

```
if q1 == memory          → bereaved (Q2 무시, 예외)
elif q2 == loss          → bereaved
elif q2 == care          → caregiver
elif q2 == health        → self_prep
elif q2 == milestone     → reflector
else (q2 == none):
    q1 == care    → caregiver
    q1 == pre     → self_prep
    q1 == reflect → reflector
    else          → explorer
```

stage는 5종: `caregiver / self_prep / reflector / bereaved / explorer` + `solo` 플래그(Q3=alone).

## 1. 케이스 매트릭스 — Q1 × Q2 (25 케이스)

표기: `→ stage / 정합성(평가) / 비고`. 평가는 **사용자가 답하면서 어색/모순/허위로 느낄 가능성**.

| Q1 ↓ \\ Q2 → | care(편찮음) | loss(보내드림) | health(검진) | milestone(은퇴·손주) | none(계기 없음) |
|---|---|---|---|---|---|
| **reflect**(돌아봄) | caregiver / **△ 모순** : 돌아보고 싶다는데 결과는 케어링 | bereaved / **△ 모순** : 회고 의도가 사별로 덮임 | self_prep / **△ 모순** : 회고 의도가 자기준비로 덮임 | reflector / 정합 | reflector / 정합 |
| **pre**(앞일 챙김) | caregiver / **△ 의도 잃음** : 자기준비 의도가 케어링으로 | bereaved / **△ 모순** | self_prep / 정합 | reflector / **△ 모순** : 자기준비 → 회고로 둔갑 | self_prep / 정합 |
| **care**(돌봄 중) | caregiver / 정합 | bereaved / **△ 시제 충돌** : 돌보는 중 + 보내드림 동시? | self_prep / **△ 모순** : 돌봄 중인데 결과는 자기준비 | reflector / **△ 모순** | caregiver / 정합 |
| **memory**(사별) | bereaved / **❌ Q2 옵션 부적합** : "편찮으세요"는 현재형 → 사별 후 답할 게 없음 (창업자 케이스) | bereaved / **❌ 중복** : Q1과 같은 의미를 두 번 묻는 꼴 | bereaved / **❌ 무관** : 사별과 본인 검진 무관 | bereaved / **❌ 무관** | bereaved / △ : "특별한 계기 없음"인데 사별이 계기. 형식상만 통함 |
| **explore**(모름) | caregiver / **△ 무리한 path 확정** : 모른다는데 케어링 판정 | bereaved / 정합 | self_prep / 정합 | reflector / 정합 | explorer / 정합 |

### 1.1 카운트
- 정합(자연스러움): **10/25 = 40%**
- 모순·미스매치·중복: **15/25 = 60%**
- 그중 사장님 케이스(memory 행)는 **5칸 중 5칸 모두 어색**.

## 2. 핵심 진단 (라벨 7개)

1. **[D1] 사별 path 빈자리** — Q1=memory를 선택한 사용자에게 Q2 5개 옵션은 모두 부적합. care/health/milestone은 무관, loss는 Q1과 동의어 반복, none은 거짓말 강요. 사장님 케이스의 정체.
2. **[D2] Q1·Q2 의미 중첩** — Q1은 "마음이 머무는 곳"(상태), Q2는 "계기"(원인)인데, 옵션 어휘는 둘 다 lifecycle 단어를 쓰고 있어 사용자는 같은 질문을 두 번 받는 느낌. 특히 `q1=care` × `q2=care`(돌봄중+편찮음), `q1=memory` × `q2=loss`(사별+보내드림)는 동의어 반복.
3. **[D3] 의도 덮어쓰기** — Q2 우선 로직(L201–207) 때문에 Q1에서 "회고/자기준비"를 골라도 Q2가 다른 trigger면 stage가 바뀐다. 사용자는 자기가 고른 1번이 무시당했다는 인상.
4. **[D4] 분기 없음** — 25칸 중 15칸이 어색한 근본 원인은 lifecycle path별 분기가 없다는 것. 사별·돌봄·자기준비는 살아가는 시간축이 다른데 같은 Q2를 묻는 게 구조적 오류.
5. **[D5] Q4·Q5·Q6 dead weight** — 코드 확인 결과 Q4(마음 무게), Q5(요즘 생각), Q6(시작 강도)는 **결과 화면에서 한 번도 사용되지 않음.** 저장만 됨. 7문항 중 3개가 사용자 시간만 잡아먹는 셈. 마케팅 관점에서 가장 큰 이탈 위험.
6. **[D6] Q3 무게 동일 처리** — 사별 후 "누구와 함께"와 케어링 중 "누구와 함께"는 다른 무게의 질문인데(전자는 "곁에 누가 있나"의 위로, 후자는 "역할 분담"의 실무) 같은 옵션·같은 결과 카피(SOLO 블록 하나)로 처리.
7. **[D7] 결과 부재한 솔루션 광고** — 5/26 확정 문서에는 인터스티셜·결과·CTA가 풍부하지만, 사용자 입장에선 Q4–Q6의 답이 결과에 반영 안 됨 → "왜 물었지?" 인지부조화. 이탈 페르소나(특히 사별 직후, 인지자원 적음)에 치명적.

## 3. 대안 비교 (옵션 A / B / C)

### 옵션 A — lifecycle 분기 트리 (4–5문항, path별 분기)
- **구조:** Q1에서 lifecycle 5 path 확정 → path별로 다음 2–3 질문이 다르게 나옴 → Q마지막에 강도(read/answer/do) → 결과.
- **질문 수:** 평균 4–5 (path별 가변).
- **장점:** 정합성 위험 거의 사라짐(D1·D2·D3·D4·D6 해결). 사별 path에는 사별 맥락 질문만 보임.
- **단점:** 구현 복잡도 ↑ (path별 STEPS 분기 배열 + computeStage 단순화). 카피 작업량 ↑ (path × question 매트릭스).
- **acquisition 영향:** 이탈 ↓ (관련성 ↑), 완료율 ↑ 예상. 결과 카피의 개인화 깊이 ↑.
- **결과 화면:** 답한 항목이 모두 결과에 반영됨 → "왜 물었지?" 사라짐.

### 옵션 B — 통합 단순화 (3문항, 분기 없음)
- **구조:** Q1=lifecycle path 직접 선택(현 Q1 + Q2 통합한 단일 명확 질문). Q2=강도(read/answer/do). Q3=한 문장(현 Q7). 끝.
- **질문 수:** 3.
- **장점:** 가장 단순. 사장님 케이스의 어색함 자체가 사라짐. 7문항 → 3문항이면 완료율 큰 폭 상승 예상.
- **단점:** Q3(누구와)·Q4(마음 무게) 등 미러링·SOLO 분기 정보가 사라짐 → 결과 카피의 따뜻함·개인화가 얕아짐. "1분이면 돼요"가 아니라 "10초면 돼요"인데, 가입 전 감정 빌드업이 부족.
- **acquisition 영향:** 완료율 ↑↑, 가입전환률은 양면. 짧으니 페르소나 깊이 빌드업 부족.

### 옵션 C — 진단 줄이고 콘텐츠로 흘리기 (1문항 + 즉시 추천)
- **구조:** Q1만 (lifecycle 5 path) → 결과 화면 없이 path별 추천 콘텐츠 3편 + 도구 1개 노출. 가입은 콘텐츠 읽다가 "저장" 액션에서.
- **질문 수:** 1.
- **장점:** 퍼널 마찰 최소. acquisition 관점에선 최고. 终活ねっと가 놓친 "콘텐츠로 머무는 이유"에 가장 가까움.
- **단점:** 미러링·동행 한 줄·"내 자리가 그려졌어요"의 감정 클라이맥스가 사라짐 → 잇다 브랜드 톤의 핵심 자산을 버리는 셈. 가입게이트 위치 재설계 필수.
- **acquisition 영향:** 콘텐츠 클릭률 ↑, 가입전환률 ↓ 가능성. 잇다의 차별점인 "조용한 동반자" 인상도 약화.

### 비교표

| 축 | A (분기 트리) | B (3문항 통합) | C (1문항 + 콘텐츠) |
|---|---|---|---|
| 질문 수 | 4–5 (가변) | 3 | 1 |
| 시간 | 45초 | 20초 | 5초 |
| 정합성 위험 | 매우 낮음 | 매우 낮음 | 매우 낮음 |
| 완료율 | 중·상 | 상 | 최상 |
| 가입 전환 깊이 | 상 (개인화 ↑) | 중 | 하 |
| 결과 카피 풍부도 | 최상 | 중 | 하 (결과 화면 없음) |
| 브랜드 톤(미러링) | 잘 살림 | 일부 살림 | 잃음 |
| 구현 복잡도 | 상 | 하 | 하 |

## 4. 추천 — 옵션 A (lifecycle 분기 트리)

### 4.1 추천 근거
- D1–D7 진단의 근본 원인이 **분기 부재**. 옵션 B/C로는 D5(Q4–Q6 dead weight)만 해결되고 잇다 톤의 핵심(미러링·동행)이 같이 깎인다.
- 5/26 카피 확정 작업의 자산(인터스티셜·동행 한 줄·SOLO 블록·Q7 미러링)을 그대로 살리면서 정합성만 잡는 길이 옵션 A.
- 단, 완료율 보호를 위해 **path별 질문 수를 3–4로 제한**, 진행률 표시도 path 결정 후 동적으로 업데이트.

### 4.2 새 질문·옵션 카피 안 (잇다 톤)

#### Q1 (공통, single) — path 결정
**지금, 어디에 가장 마음이 머무세요?**
- 곁에서 누군가를 돌보고 있어요 → `caregiver`
- 앞일을 미리 챙겨두고 싶어요 → `self_prep`
- 살아온 시간을 한번 돌아보고 싶어요 → `reflector`
- 소중한 분을 떠나보냈어요 → `bereaved`
- 아직 잘 모르겠어요 → `explorer`

> Q1만으로 path 확정. 기존 Q2의 "계기" 질문은 path별로 다르게 묻는다 → 아래.

#### Q2 (path별, single) — 계기·상황 (path 맞춤)

**caregiver path:**
**지금 돌보고 계신 분은 어떤 상황이세요?**
- 입원이나 시설에 계세요
- 댁에서 함께 돌보고 있어요
- 멀리 떨어져 챙기고 있어요
- 막 시작한 참이에요

**self_prep path:**
**미리 챙겨두고 싶다는 마음, 어디서 시작되었어요?**
- 건강검진이나 진단에서 마음에 걸리는 게 있었어요
- 은퇴·환갑·손주처럼 삶의 매듭을 지났어요
- 부모님을 보내드리며 나도 정리해 둬야겠다 싶었어요
- 특별한 계기는 없어요. 그냥 한번 챙겨보려고요

**reflector path:**
**돌아보고 싶다는 마음, 어떤 결로 와 있어요?**
- 글로 남겨두고 싶어요
- 자식·손주에게 들려주고 싶어요
- 나만 알던 이야기, 누가 들어줬으면 해요
- 아직 모르겠지만, 일단 적어보고 싶어요

**bereaved path:** (사장님 케이스 — 빈자리 메움)
**보내드린 지, 어느 정도 되셨어요?**
- 얼마 안 됐어요
- 몇 달, 또는 한 해쯤 됐어요
- 시간은 꽤 지났는데, 마음은 아직이에요
- 말로 표현하기 어려워요

**explorer path:**
**오늘은 어떤 결로 들르셨어요?**
- 가족 일로 생각이 많아져서요
- 콘텐츠 한 편 보러요
- 그냥 이런 곳이 있다고 들어서요
- 잘 모르겠어요

#### Q3 (공통, single) — 누구와
**이 마음, 지금 누구와 나누고 계세요?**
- 형제·자매와요
- 배우자와요
- 자녀와요
- 지금은 혼자예요

> bereaved path에서는 Q3 카피만 살짝 변주 가능: "지금 곁에 누가 있어요?" — 동행의 의미가 "함께 정리"가 아니라 "곁에 있음"으로 바뀜.

#### Q4 (공통, single) — 강도 (현 Q6 대체)
**지금 시작한다면, 어느 정도가 편하시겠어요?**
- 짧은 글 하나 읽는 정도
- 질문 하나에 답해보는 정도
- 뭔가 하나 정리해서 결과물을 보고 싶어요

#### Q5 (공통, text, 선택) — 한 문장 (현 Q7 유지)
변경 없음.

#### 제거
- 기존 Q4(마음 무게 heavy/mid/light) — 결과에 반영 안 됨, 제거.
- 기존 Q5(요즘 생각 multi) — 결과에 반영 안 됨, 제거. 단 향후 콘텐츠 개인화 추천 도입 시 재도입 검토 여지.

### 4.3 결과 화면 — Q2 path별 답 반영
- caregiver path에서 Q2=`멀리 떨어져 챙기고 있어요` 답이면 동행 한 줄에 "멀리서도 같은 화면을 보면 곁에 있는 것과 다르지 않아요" 변주.
- bereaved path에서 Q2=`얼마 안 됐어요` 답이면 첫걸음 CTA를 **`도구 강제 X`**, "오늘은 그냥 글 한 편만 읽고 가셔도 돼요"로 변주(5/26 확정문서의 사별 path 가드와 일치).
- self_prep path에서 Q2=`부모님을 보내드리며…` 답이면 사별 path 콘텐츠도 함께 추천(이중 path 미러링).

### 4.4 진행률 표시
