## 0. 결론 요약 (사장님 3분 독해용)

1. **묻다 콘텐츠 = "읽을거리"가 아니라 "매듭에 붙는 실천 가이드"다.**
   잇다 정보 허브(검색에 답하는 백과)와 역할을 나눈다. 묻다에서 콘텐츠는 목적지가 아니라
   **도구로 가는 경사로** — 행동의 문턱(두려움·막막함)을 낮춰 도구 완료율을 올리는 장치.
   볼륨 게임(SEO·연재)은 브런치와 잇다가 하고, 묻다는 **매듭당 1편, 총 7~9편으로 캡**한다.

2. **배치 추천: B안(여정 매듭 인라인) 주축 + A안(길잡이 = 서가·외부 랜딩) 보조.**
   홈 피드(D안)는 기각 — 묻다의 "오늘의 한 걸음 하나" 설계와 조용함 원칙을 깬다.

3. **치명적 결함 발견: 묻다 아티클이 현재 로그인 필수다** (`article-*.html`이
   `initAppPage()` 호출 → 비로그인 리다이렉트). 브런치 독자가 글을 타고 와도 읽지 못하고
   가입 장벽부터 만난다. **아티클 비회원 공개 전환이 P0.**

4. **정합성 결함 2: 브런치 원고 9편의 CTA가 전부 잇다로 간다** (6/21 전략 = 투 브랜드
   결정 7/6 이전). "비밀번호 목록"·"못 깨어난다면" 같은 자기 준비 글의 CTA를 묻다로
   재정렬해야 한다. 특히 09번 글의 목적지였던 잇다 will-start는 이미 묻다로 리다이렉트
   스텁 처리된 상태(7/8) — 지금 발행하면 링크가 두 번 튄다.

5. **분류 기준 = 판정 질문 3개**: ①주어가 '나'인가 '부모/가족'인가 ②시제가 '미리'인가
   '닥침/이후'인가 ③자연스러운 CTA가 묻다 도구인가 잇다 도구인가. 2개 이상이면 그쪽 소속.
   **"한 시간, 종이 한 장" = 묻다** (나/미리/유품·디지털 정리). 정책은 이관·중복게시가
   아니라 **"정본 1곳 + 형식 변환"** — 에세이 원본은 브런치(획득 채널), 앱에는 같은 소재의
   실천 가이드판.

6. **Q3 기여 경로**: 브런치 ref 태깅 가입(1,000명 목표의 8월 주력 채널) + 가이드→도구
   전환(유언장 초안 20%·D7 25% 목표 지원). 콘텐츠 KPI는 조회수가 아니라
   **"가이드 조회 → 도구 착수" 전환율**로 잡는다.

**창업자 결정 필요 3건**: ①기게시 브런치 글의 CTA 수정(외부 발송 영역) ②사전연명의향서
콘텐츠·도구의 묻다 이관 여부 ③잇다 `well-dying-guide` 이관 vs 브리지 (각 §4·§6 참조).

## 1. 역할 정의 — 잇다 콘텐츠와 무엇이 다른가

### 1-1. 두 서비스의 콘텐츠 분업

| | **잇다 콘텐츠** (info/* + 오늘잇고 에세이) | **묻다 콘텐츠** (길잡이) |
|---|---|---|
| 독자의 상황 | 문제가 **닥쳤다** — 요양·호스피스·사별 절차·상속 처리 | 문제가 **오기 전이다** — 준비를 시작할까 말까 망설임 |
| 역할 | **검색에 답하는 백과** — SEO 입구, 정보 완결성 | **행동에 붙는 길잡이** — 도구 사용의 문턱 낮추기 |
| 성공 지표 | 검색 유입·체류 | **가이드 조회 → 도구 착수 전환율** |
| 볼륨 전략 | 계속 확장 (검색어 커버리지) | **매듭당 1편, 7~9편 고정** — 늘리지 않는 게 전략 |
| 형식 | 가이드·에세이·체크리스트 다양 | 아래 "매듭 가이드" 단일 포맷 |

**왜 이렇게 나누나**: 묻다의 리텐션 기둥은 콘텐츠가 아니라 ①오늘의 질문 ②안부확인이다
(기획서 §6). 콘텐츠로 리텐션을 만들려 하면 잇다 정보 허브의 복제품이 되고, 슈카츠넷의
"읽고 끝나는 트래픽" 함정을 앱 안에 축소 재현하게 된다. 묻다 콘텐츠의 존재 이유는 단 하나 —
**읽은 사람이 5분 뒤 도구를 시작하게 만드는 것.**

### 1-2. "매듭 가이드" 표준 포맷 (제안)

각 여정 매듭(journey.js의 7 STEPS)에 1:1로 붙는 글. 5분 이내.

```
① 왜 (2문단)         — 1인칭 훅. 창업자 경험 or 먼저 한 사람의 목소리
② 미리 알아둘 사실     — 팩트 3~5개. YMYL이면 감수 표기 필수
③ 지금 하기 (CTA)     — "이 글의 체크리스트를 내 것으로" → 해당 도구 딥링크
④ 자주 묻는 걱정 3개   — "가족이 이상하게 보면?" 류의 심리 장벽 해소
```

기존 2편이 이미 이 방향과 맞다: `article-will-legal`(유언장 매듭 짝, 감수 표기 있음),
`article-belongings`(유품 매듭 짝). **새로 만드는 게 아니라 이 패턴을 7개로 완성**하는 일이다.

## 2. User Journey Map — 콘텐츠가 끼는 8개 지점

전제 유입 3경로: **A. 브런치** (고맥락 — 이미 3분 에세이로 신뢰 형성됨) ·
**B. 인스타 자동DM** (`/mutda/?ref=beta_ig`, 저맥락 호기심) · **C. 검색** (미래 — 공개
아티클이 인덱싱되면).

| 단계 | 사용자 상태 | 화면/터치포인트 | 콘텐츠의 역할 | 다음 행동 트리거 | 계측 |
|---|---|---|---|---|---|
| **1. 발견** | 브런치 글/카드뉴스 읽는 중 | 브런치·인스타 (외부) | 에세이 = 공감·신뢰 형성 | 글 말미 CTA 1개 (§5) | 브런치 통계 · ManyChat |
| **2. 소프트 랜딩** | "나도 해볼까" 반신반의 | **묻다 공개 아티클**(주제 페어) 또는 랜딩 `?ref=brunch_*` | 가이드 = "생각보다 간단하네"로 번역. **가입 요구 전에 읽게 한다** | 하단 "이 목록을 내 것으로 만들기" | `article_view`(비회원 포함) + ref |
| **3. 가입·온보딩** | 시작 결심 | signup → 퀴즈 4문항 | (콘텐츠 없음) 유입 주제가 `journey_focus` 앞배치에 반영 | 첫 질문 답 = 첫 성공 | `signup_done` `onboarding_done` |
| **4. 첫 실천 (aha)** | 도구 첫 화면, 막막함 재발 지점 | 도구 화면 상단 **접힌 인라인 가이드** "시작 전 3분 읽기" | 심리 장벽 해소 (④걱정 3개) | 품목 1개 등록 / 편지 1통 / 유언 1문항 | `belonging_added` 등 |
| **5. 매듭 완료 순간** | 성취감 — 가장 수용적인 순간 | 완료 화면/토스트 아래 **"다음 매듭 가이드" 카드** | 여정의 접착제 — "다음은 왜 이것인지" 3분 | 다음 매듭 진입 | step_done → `article_view` 연쇄 |
| **6. 데일리 복귀** | 습관 형성 중 | 홈: 오늘의 한 걸음 + 오늘의 질문. **"함께 읽어보세요" = 내 다음 매듭의 가이드로 동적화** (현재 하드코딩 2편) | 오늘의 걸음이 무겁게 느껴질 때의 대체 행동 | 질문 답 / 걸음 실행 | `home_view`, D7 |
| **7. 정체·이탈 구간** | 3~7일 무활동 | (v2: 웹푸시) 재방문 시 홈 카드 | **가장 가벼운 글 1편**으로 재개입 — "오늘은 읽기만 해도 돼요" | 아티클 → 도구 복귀 | 리텐션 리포트 |
| **8. 확장·바이럴** | 기록이 자산이 된 상태 | 편지 전하기(`?from=letter`) · 질문 초대(`ask.html`) · **아티클 하단 공유 버튼** | 공개 아티클 = 회원이 지인에게 보낼 수 있는 유일한 "가입 없이 읽히는" 지면 | 수신자 유입 (2로 회귀) | `sent_at`, invite, ref |

**핵심 설계 원칙**: 콘텐츠는 2(들어올 때)·4(막힐 때)·5(넘어갈 때)에만 낀다.
6·7은 보조. 데일리 훅은 어디까지나 오늘의 질문·안부확인이 담당하고, 콘텐츠는 그 자리를
넘보지 않는다.

### 유입 경로별 여정 예시 — "한 시간, 종이 한 장" 독자

```
브런치 "유서가 아니라, 비밀번호 목록을 먼저 썼습니다" (3분)
  → 말미 CTA: "그 종이 한 장, 묻다에서 만듭니다"
  → mutda/article-digital-onepage.html?ref=brunch_password  [비회원 공개]
     (브런치 = 왜 / 묻다 가이드 = 어떻게: 은행·보험·잠금번호·구독료 항목별 요령)
  → 하단 CTA "이 목록을 내 것으로" → signup (ref 유지)
  → 온보딩 → 유품 매듭이 여정 앞으로 → 디지털 품목 1개 등록 (aha)
  → 완료 카드: "다음 매듭 — 감사의 말. 왜인지 3분" → 편지 → 오늘의 질문 데일리 정착
```

## 3. 배치안 비교 — 어디에 콘텐츠를 두나

| 옵션 | 내용 | 장점 | 리스크 | 공수 |
|---|---|---|---|---|
| **A. 길잡이 탭 확장** | 카테고리별 서가로 키움 (목록·태그) | 구조 단순 · 비회원 SEO 랜딩 · 이미 있는 탭 | 단독이면 "읽고 끝" — 도구와 분리 (슈카츠넷 축소판). 잇다 허브와 중복 시작점 | 소 (0.5일) |
| **B. 여정 매듭 인라인** | 도구 진입 화면(접힌 3분 읽기) + 매듭 완료 화면(다음 매듭 가이드 카드) | **행동 전환에 직결** · 여정 서사 일관 · 조용함 유지 | 발견성 낮음(브라우징 불가) · 도구 화면 복잡화 위험 → 기본 접힘으로 해소 | 중 (1~1.5일) |
| **C. 오늘의 한 걸음에 읽기 태스크** | 행동 매듭 사이에 "오늘은 이 글 읽기" 걸음 삽입 | 데일리 소비 리듬 · 가벼운 날 제공 | 여정=행동 원칙 희석("숙제" 느낌) · 매듭 완료 정의 복잡 · 7매듭이 12걸음으로 늘며 완주감 저하 | 중 (1일+여정 로직 변경) |
| **D. 홈 콘텐츠 피드** | 잇다식 '요즘 이야기' 피드 | 콘텐츠 소비 극대화 | **기각** — "한 걸음 하나" 단일 초점 파괴 · 잇다와 정체성 중복 · 7편으로 피드 성립 불가 | 대 |

### 추천: **B 주축 + A 보조** (C는 8월 데이터 후 재검토, D 기각)

- **B가 주축인 이유**: §1의 역할 정의(콘텐츠 = 도구로 가는 경사로)를 그대로 구조화한 것.
  Q3 목표 지표(유언장 초안 20%, D7 25%)에 직접 기여하는 배치는 B뿐이다.
- **A의 역할 재정의**: 길잡이 탭은 "확장"이 아니라 **①7~9편의 전체 서가(회원 브라우징)
  ②비회원 공개 랜딩(브런치·검색·공유 유입의 착지면) ③잇다 브리지 유지**. 지금 구조에서
  바꿀 것은 아티클 공개 전환과 목록 매니페스트뿐.
- **가장 큰 trade-off**: B는 콘텐츠 조회수가 낮게 나온다. 이건 실패가 아니라 설계 의도 —
  콘텐츠 KPI를 조회가 아니라 **가이드 조회→도구 착수 전환율**로 잡아야 하는 이유다.
  [가설] 전환율 30%면 성공, 10% 미만이면 가이드 무용론 재검토.
- **반대 견해 (기록)**: "Q3는 리텐션 검증이 전부인데 콘텐츠 7편 제작은 분산이다.
  콘텐츠는 전부 브런치(외부)에 몰고 묻다는 도구만 다듬어야 한다"는 시각이 있을 수 있다.
  일리 있으나, 아티클 비공개 상태로는 브런치 유입이 가입 장벽에 직행하는 현 구조가 더 큰
  누수다. 절충이 "7~9편 캡 + 매듭 가이드 포맷 고정" — 콘텐츠 조직을 만들자는 게 아니라
  깔때기의 이 빠진 구간을 메우자는 제안이다.

## 4. 잇다 ↔ 묻다 콘텐츠 분류 기준

### 4-1. 판정 질문 3개 (2개 이상 해당하는 쪽 소속)

1. **주어**: 이 글의 주인공이 '나'인가(→묻다) '부모·남은 가족'인가(→잇다)?
2. **시제**: '미리 준비'인가(→묻다) '닥친 일·이후 수습'인가(→잇다)?
3. **CTA**: 글을 읽고 자연스러운 다음 행동이 묻다 도구(유언장·편지·유품·안부)인가,
   잇다 도구(케어 일지·요양 검색·절차 체크리스트)인가?

### 4-3. 이관·공유·중복게시 정책 (제안)

1. **정본 1곳 원칙.** 같은 글을 잇다·묻다 양쪽에 게시하지 않는다 (유지보수 2배 + 검색
   신호 분산 + "어느 앱 글인가" 정체성 혼선).
2. **에세이는 채널이 원본.** 1인칭 에세이("한 시간, 종이 한 장")의 정본은 브런치(획득
   채널). 앱 안에는 같은 소재의 **실천 가이드판**(형식 변환)을 둔다 — 이건 중복이 아니라
   분업이다. 브런치 = 왜, 묻다 = 어떻게.
3. **URL 이동 시 리다이렉트 스텁.** 잇다 유언장 3페이지에서 이미 검증된 패턴(7/8 worklog).
   이관 결정이 나면 같은 방식으로.
4. **경계 콘텐츠는 이관보다 브리지 먼저.** `well-dying-guide` 등은 글 하단에 묻다 브리지
   카드("이 준비, 묻다에서 시작할 수 있어요")를 먼저 달고, 유입 데이터를 보고 이관을
   판단한다. 지금 옮기는 건 데이터 없는 직감 베팅.
5. **잇다 상속 페이지들에 묻다 유언장 브리지.** 사장님 확인 데이터 = 돈·상속 검색 유입이
   압도적. 그 독자 중 "미리 설계" 니즈(생전 유언·증여)는 묻다 고객이다. 잇다에서 가장 큰
   유입 지면에 묻다로 가는 문이 없는 게 현재 누수.

## 5. 브런치 → 묻다 유입 동선 (CTA 설계)

### 5-1. 링크·태깅 규약

```
https://suechoi033-private.github.io/life-heritage-v2/mutda/{목적지}?ref=brunch_{campaign}
```

- **`?ref=`가 주 파라미터** — `mutda/js/referral.js`가 first-touch로 localStorage 캡처 →
  가입 시 `profiles.referral_source` 정착 → 대시보드에서 **가입 어트리뷰션**까지 추적됨
  (기구현, 7/7). UTM은 web_events 방문 로그용으로 병기 가능하나 링크가 길어지므로
  ref 단일을 기본으로 제안.
- 값 컨벤션: `brunch_password` · `brunch_will` · `brunch_farewell` · `brunch_inheritance`
  (인스타 기존 `beta_ig`와 나란히 대시보드 유입경로 분류에 잡힘).
- **목적지 원칙**: 페어 가이드가 있으면 **공개 아티클**(소프트 랜딩), 없으면 랜딩
  `mutda/`. 도구 딥링크(will.html 등)는 비로그인 시 로그인 화면으로 튕기므로 금지.

### 5-3. CTA 문구 템플릿 (톤: 재촉 없음, 말미 1개)

> ―― 그 종이 한 장을 만드는 곳, **묻다**를 만들고 있습니다.
> 질문에 하나씩 답하면 나의 정리 목록이 만들어집니다. [10분이면 시작할 수 있어요 →]

- 기존 규칙 유지: 글당 링크 1개, 본문 아닌 말미, 다음 글 예고 + 구독 유도와 병치.
- 인스타 카드뉴스 6장 CTA(마지막 장)도 동일 재정렬 — "전문은 브런치에서"는 유지하되
  브런치 글의 CTA가 묻다를 가리키는 2단 구조.

이미 발행된 브런치 글에 잇다 CTA가 실려 있다면(예: 첫 글, 1~2주차 발행분) 수정은

불일치)은 게시 전/후 교체를 권한다.
