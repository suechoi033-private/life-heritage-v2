## 1. 사장님 통찰 정리 (전문 + 해석)

### 1.1 사장님 발화 전문 보존

> "마케팅을 위한 카드뉴스를 보니, 우리 서비스에 적용할 것이 눈에 보이는데? 사실 이걸 해줘야 할 것 같아. 작은 연락을 하게끔. 다시 못 깨어난다면? 거의 언제나 사람을 떠올려. 그럼 누구? 그에게 못한 말? 하고 싶은 말? 내일 다시 못 깨어난다면? 이라는 질문을 던지고, 떠오르는 사람을 적고, 그에게 주 몇 회 연락할건지 쓰게 하고, 그걸 하게끔 행동의 넛지를 주는 역할을 우리 프로덕트에 포함해보자. 사소한 기쁨도 마찬가지. 나를 위한 풍요로운 삶이라는 코너가 있고, 내가 원하는 게 뭔지 쓰게 하자. 이게 어떻게 보면 버킷리스트에 포함해둔 건데 이게 일기 쓰는 데에 같이 합해져 있어서 드러나지가 않아. 일기를 드러내도 되지 않을지 팀과 상의해봐. 자기성찰은 이미 잘 되어 있고."
>
> "**프로덕트의 축을:**
> 1. **내가 내일 다시 못 깨어난다면? — 떠오르는 사람, 꼭 하고 싶은 일을 쓰게 하고 그걸 행동으로 연결할 수 있도록 해준다.**
> 2. **나를 위한 풍요로운 삶. 내가 원하는 거.** 이걸 '다시 못 깨어난다면'의 또 하나의 축으로 하위메뉴로 볼 수도 있을 것 같은데 여튼 이것도 쓰게 하고 이것도 달성할 수 있게 도와주자.
> 케어링과 자기성찰은 그냥 두고."

### 1.2 한 줄 해석

사장님은 seed-12에서 카드 1·3·4·6(질문 → 떠오른 사람 → 4개 후회 카테고리 → 3개 행동)을 **앱의 골격으로 옮기자**고 하셨다. 즉 콘텐츠가 환기시킨 "오늘 해야 할 일"을 사용자가 **글로 쓰고, 행동으로 연결**할 자리를 앱 안에 만든다.

### 1.3 두 가지 결이 동시에 있다

| 결 | 뿌리 (seed-12 카드) | 잇다에서 비어 있는 자리 |
|---|---|---|
| **A. 사람** ("거의 언제나 사람") | 카드 3·4 (0.5초 만에 떠오른 그 한 사람 / 표현하지 못한 마음 + 닿지 못한 관계) + 카드 6의 1번 항목 (떠오른 그 한 사람에게 오늘 연락하기) | 현재 잇다의 어디에도 "내 삶에서 중요한 사람"이라는 **명시적 객체**가 없다. 케어링(nest)은 부모/조부모처럼 돌봄 대상에 한정. |
| **B. 사소한 기쁨** ("나를 위한 풍요로운 삶") | 카드 4의 4번째 카테고리 (미뤄둔 사소한 기쁨) + 카드 6의 2번 항목 (미뤄둔 사소한 기쁨 하나를 이번 주에 해보기) | 현재 잇다의 버킷리스트(`goals`)는 영역 4개 — 재정·건강·가족·개인성장. **"사소한 기쁨"·"오늘 하고 싶은 일"이 들어갈 자리가 명시적으로 없다.** 영역 4개는 모두 "장기 목표" 결. |

### 1.4 사장님이 명시한 비대상 (건드리지 말 것)

- **자기성찰**(ask.html / 오늘 잇고): "이미 잘 되어 있고."
- **케어링**(nest.html): "그냥 두고."

→ 본 라운드의 변경 범위는 **"라이프"(seed.html) 탭 내부**로 한정된다.

### 2.2 BM 본질(D5 보류) 신호와의 정합

`docs/strategy/decisions-2026-06-13.md` 3장에서 사장님이 던진 신호: **"구독료 형태로는 못 갈 것 같다"** → BM 무게가 "도구 유료화 + 매칭"으로 이동할 수 있음.

새 축은 이 신호와 정합 가능한가?

- **task-oriented 결제와 부합**: Axis B(풍요로운 삶) 항목 중 "회고록 책 출판"·"AI 정리" 같은 **완성물 결제**가 본질적으로 자기 표현 결. 사장님의 풍요로운 삶 코너에 "이번 분기 회고록 한 권" 같은 항목이 들어오면 도구 유료화 1축의 직접 발화점이 된다.
- **단, 새 축 자체는 무료가 정답**: Axis A(중요한 사람 + 연락 넛지)는 신뢰·관계 영역 — 여기서 결제 트리거를 걸면 사용자가 부담을 느낀다(헌장 톤 위배). 결제는 "완성물"(회고록·인쇄·캡슐화) 단계에서만.
- **"task 시점 결제"의 진정한 자리**: Axis A에서 알림·기록 자체는 무료 / 분기마다 "지난 3개월 ○○에게 보낸 메시지 모음 → 인쇄 카드/디지털 캡슐" 같은 **결과물 결제** 가능성. 본 라운드 결정 대상은 아님(별도 BM 라운드).

## 3. 사실관계 (코드 확인)

### 3.1 일기

| 항목 | 사실 |
|---|---|
| 진입점 | `seed.html` 탭1(기본값=활성) → "일기 쓰기" CTA → `diary-write.html`. nav의 "라이프" 탭이 `seed.html?tab=diary`로 들어옴. nav 상단의 빠른 쓰기 ✏️ 아이콘도 일기 쓰기로 직결. |
| 데이터 모델 | `diary_entries` 테이블. 컬럼: `id`, `user_id`, `entry_date`, `title`, `content`, `template_type`(`TEMPLATES`로 유형 선택), 미디어. `diary_goal_links`로 일기와 buckletlist 연결 가능. |
| 진행 흐름 | 캘린더(달별)·타임라인(스크롤) 두 가지 노출. 한 문장 공유 카드 모달 존재. |
| 노출 위치 | `seed.html` 첫 화면(`tab=diary` 기본값). |

### 3.2 버킷리스트 ("사장님 표현: 일기에 묻혀 있다")

| 항목 | 사실 |
|---|---|
| 진입점 | `seed.html` **두 번째 탭** ("버킷리스트") → "＋ 버킷에 담기" → `plan-write.html`. nav 어디에도 직접 진입 없음. |
| 데이터 모델 | `goals` 테이블. 영역(`area`) 4개: **finance(재정) / health(건강) / family(가족) / growth(개인 성장)**. 기간(`period_months`): 3·6·12·60개월. 우선순위, 진행률, 상태. 세부계획 `goal_plans` 별도 테이블. |
| 진행 흐름 | 버킷리스트 → 영역 선택 → 제목·설명 → 기간 → 우선순위 → 세부 plan. 진행률(`progress_pct`) 자동 계산. |
| 노출 위치 | `seed.html?tab=goal` 으로 가야만 보임. **회원 홈(`index.html`) 어디에도 버킷리스트 카드 노출 없음.** |

**"일기에 묻혀 있다"의 정확한 의미 — 코드 확인 결과**:

- 테이블은 분리(`diary_entries` 와 `goals`).
- **묻힘은 진입 동선**: seed.html 진입 시 기본 탭이 일기. 탭 두 개 중 두 번째라 시각상 묻힘.
- nav 라벨도 "라이프"(seed.html) — 사용자가 "버킷리스트"를 찾으려면 nav→라이프→탭 전환 2단계 필요.
- **회원 홈에 버킷리스트 카드 0건**. 회원 홈은 (1) 오늘 잇고(ask.html) + (2) 장례희망(ceremony.html) 두 카드만.
- 영역 4개에 **"관계"·"사람"·"사소한 기쁨"이 명시 없음**. growth("개인 성장")가 가장 가까우나 추상적이라 사장님이 말한 "내가 원하는 거"·"미뤄둔 사소한 기쁨"이 정확히 들어가지 않는다.

→ 사장님 직관("버킷리스트가 묻혀 있다")은 **부분적으로 맞다**. 테이블 통합은 아니지만 (a) 진입 동선이 2단계 (b) 영역 분류가 사장님 통찰 결과 안 맞음 (c) 홈 노출 0 — 세 가지가 합쳐져 사용자 인지상 "일기 옆에 있는 무언가" 정도로 머문다.

→ 사장님 "이미 잘 되어 있고" 평가에 일치. 변경 대상 아님.

### 3.5 알림 인프라

- Edge function `push-notify` 존재. 케어링 업데이트 알림에 이미 사용 중.
- 즉 **새 푸시 알림 채널을 0에서 구축할 필요 없음**. 기존 push 인프라에 Axis A의 "주 N회 연락 nudge"를 동승시킬 수 있다.

### 3.6 코드 확인에서 가장 놀란 점 (정직히)

1. **버킷리스트는 사장님 표현보다 더 잘 만들어져 있다.** 영역·기간·진행률·세부 plan까지 모델 완비. "묻혀 있다"는 표현은 (a) 진입 동선 (b) 분류 부적합 (c) 홈 노출 0 의 합성어이지, 모델 부재가 아님. **결론: 버킷리스트를 새로 만드는 게 아니라, 동선과 분류를 재설계한다.**
2. **잇다 어디에도 "내 삶에서 중요한 사람"이라는 객체가 없다.** 케어링은 돌봄 대상(부모)뿐. Axis A는 **잇다 데이터 모델의 새 1급 객체**를 추가하는 결정이다 — 즉 one-way door 후보(DB 스키마 변경). 가벼운 변경이 아니다.

## 4. 새 프로덕트 축 — Axis A·B 설계

### 4.1 Axis A — "내일 다시 못 깨어난다면, 떠오르는 사람"

#### 4.1.1 정의

> 잇다 안에서 사용자가 **자기 삶에서 중요한 사람들의 목록**을 작성하고, 각 사람에게 **연락 빈도 목표**(주 N회, 월 N회, 분기 N회)를 정하고, **잔잔한 행동 넛지**로 그 목표를 지키도록 돕는 코너.

#### 4.1.2 진입점 옵션

| 옵션 | 설명 | 평가 |
|---|---|---|
| **A1. 가입 1주차 onboarding 직후 첫 액션** | onboarding 7문항 끝나면 곧장 "떠오르는 사람 3명을 적어볼까요?" 화면. | △ 가입 직후엔 과한 요청. 첫 5분 마찰이 D7 변종 재발 가능. |
| **A2. 라이프 탭의 세 번째 탭 (일기·버킷·**관계**)** | seed.html에 탭 추가. 일기·버킷리스트와 같은 자리. | ○ 라이프 = "내 삶을 가꾸는 자리"의 자연 확장. 사장님 표현("프로덕트의 축")과 위계 정합. |
| **A3. 회원 홈에 카드 추가** | index.html 회원 홈 카드 자리에 "이번 주 ○○에게" 카드. | ○ 매일 보는 자리 → 행동 결합 강함. 단, 회원 홈 카드 수 증가 = 다른 카드의 무게 분산. |
| **A4. 새 nav 탭** | 5탭에서 6탭으로. "관계" 별도 탭. | ✕ nav 무게 증가. 5탭이 잇다 IA 결정의 기준선(`nav.js`). |

**추천**: **A2 + A3 조합.** A2가 본진(체계적 작성·관리), A3가 매일 노출(행동 트리거). A2 없이 A3만 두면 사용자가 "누구를 적어둬야 카드가 의미가 있는지" 모름.

#### 4.1.3 입력 모델

| 필드 | 값 | 비고 |
|---|---|---|
| `name` | text, NOT NULL | 사용자 자유 입력 ("엄마", "그 친구", "이름") |
| `relation` | enum (가족·연인/배우자·친구·동료·은인·기타) | 통계·정렬용. 표시는 선택 |
| `note` | text, nullable | "왜 떠올랐는지" 자유 서술. **사장님 통찰의 "못 한 말, 하고 싶은 말"이 들어가는 자리** |
| `contact_freq_unit` | enum (week·month·quarter) | 사장님 "주 몇 회" 표현 직역 |
| `contact_freq_count` | int, 1~7 | freq_unit과 결합해 "주 2회" 등으로 |
| `priority` | int, 1~5 또는 별도 안 둠 | 사장님 표현엔 우선순위 없음 → MVP는 미도입 권장 |
| `last_contact_at` | timestamp, 사용자 수동 기록 | "오늘 연락했어요" 1탭 토글로 갱신 |
| `created_at`, `updated_at` | 자동 | |

**과시·경쟁 메커니즘 제거**:
- streak/연속 표시 ✕
- 다른 사용자와 비교 ✕
- 공개 공유 ✕ (개인 코너)
- "○일 연락 안 함" 같은 죄책감 카운터 ✕ → **대신 "이번 주 ○○에게 한 번 어떠세요"라는 잔잔한 한 줄**

#### 4.1.4 데이터 모델 — DB 스키마 제안

```sql
CREATE TABLE important_people (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  name         text NOT NULL,
  relation     text CHECK (relation IN ('family','partner','friend','colleague','mentor','other')),
  note         text,
  contact_freq_unit  text CHECK (contact_freq_unit IN ('week','month','quarter')),
  contact_freq_count int CHECK (contact_freq_count BETWEEN 1 AND 7),
  last_contact_at    timestamptz,
  archived_at        timestamptz,  -- 사별·소원해진 관계 소프트 삭제 (사별 직후 페르소나 보호)
  created_at         timestamptz DEFAULT now(),
  updated_at         timestamptz DEFAULT now()
);

CREATE INDEX idx_important_people_user_active ON important_people(user_id)
  WHERE archived_at IS NULL;

-- RLS: 본인만 SELECT/INSERT/UPDATE/DELETE
ALTER TABLE important_people ENABLE ROW LEVEL SECURITY;
CREATE POLICY important_people_own ON important_people
  FOR ALL USING (auth.uid() = user_id) WITH CHECK (auth.uid() = user_id);

-- 연락 기록 (선택, 분기 회고용)
CREATE TABLE important_people_contacts (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  person_id    uuid NOT NULL REFERENCES important_people(id) ON DELETE CASCADE,
  user_id      uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  contacted_at timestamptz DEFAULT now(),
  channel      text,  -- 'message'|'call'|'meet'|'other' — 선택
  note         text   -- 짧은 자기 메모 (Modern Loss 결: "어떤 말을 했는지")
);
```

**필터·노출 룰**:
- `archived_at IS NULL` 만 활성 목록.
- 정렬: 기본 "이번 주 목표 미달" 우선(`contact_freq` 기준 계산), 그다음 `last_contact_at` 오래된 순. 사용자 토글로 "추가한 순"·"이름 순" 변경 가능.
- 사별 케이스: archive 옵션 + "남긴 사람" 별도 view (Axis A의 슬픔 결 — 아래 7.3 시나리오 참고).

#### 4.1.5 페이지별 노출 룰

| 페이지 | 무엇이 보이나 | 어떻게 |
|---|---|---|
| `index.html` 회원 홈 | (옵션 A3 채택 시) 가장 우선순위 높은 한 명 카드: "이번 주 ○○에게 한 번 어떠세요" | `important_people` 활성 중 (이번 주 목표 - 실제 연락) 차이 최대값 1명. 0명이면 "중요한 사람 한 명 적어둘까요?" 카드. 카드 누르면 seed.html?tab=people. |
| `seed.html` 탭3 "관계" | 전체 활성 목록 + 추가 + 편집. 각 카드에 "오늘 연락했어요" 토글(=`last_contact_at` 갱신 또는 contacts insert). | 활성 목록 정렬은 4.1.4 룰. archived view는 별도 진입(작게). |
| 푸시 알림 | 주 1회(예: 일요일 저녁) "이번 주 ○○에게 한 번 어떠세요" — 활성 중 목표 미달 1명. | 옵트인 기본 OFF, 사용자 ON시만. (사장님 통찰 "행동 넛지" 직역 — 아래 6장 메커니즘 참고) |
| `nest.html` (케어링) | **연동 없음**. 케어링은 "그냥 두고" 원칙. 단 6개월+ 후 케어링 돌봄 대상(부모)를 important_people로 표시할지 별도 라운드 검토. | |

### 4.2 Axis B — "나를 위한 풍요로운 삶"

#### 4.2.1 정의

> 잇다 안에서 사용자가 **자기를 위한 작은 기쁨·해보고 싶은 일·미뤄둔 것**을 자유롭게 적고, 이번 주·이번 분기에 하나씩 실행하도록 잔잔히 돕는 코너.

#### 4.2.2 현 모델과의 관계 — 신규 vs 기존 확장

**핵심 결정**: 기존 `goals` 테이블 확장? 별도 테이블 신설?

| 옵션 | 설명 | 평가 |
|---|---|---|
| **B1. 기존 `goals` 영역에 새 영역 추가 (예: `joy`)** | finance·health·family·growth·**joy** 5개로. plan-write.html 영역 옵션에 "사소한 기쁨" 추가. | ○ 모델 변경 최소(영역 enum 추가). 진행률·기간 룰 그대로 활용. **단, "사소한 기쁨"은 기간·진행률 결이 아님** — "이번 주에 영화 한 편" 같은 단발성. |
| **B2. 별도 테이블 `joys` (또는 `wishes`)** 신설 | 가벼운 항목, 진행률·기간 없이. "이번 주 / 이번 달" 정도 시기만. | ○ "사소한 기쁨"의 본질(단발성·가벼움)에 정확히 맞음. 단, 사장님 표현 "버킷리스트에 포함해둔 건데"와 어긋남(사장님은 같은 자리로 인식). |
| **B3. `goals` 확장 + UI에서 "버킷리스트" vs "사소한 기쁨" 분리 노출** | 모델은 영역 추가(B1), UI는 두 갈래로(장기 목표 / 가벼운 기쁨). | **추천**. 사장님 직관 "버킷리스트에 포함해둔 건데 드러나지 않아"를 모델 통합 + UI 분리로 받음. one-way door 회피(영역 enum 추가는 reversible). |

**추천 = B3**.

#### 4.2.3 데이터 모델 — B3 채택 시 변경

```sql
-- goals.area enum에 'joy'(또는 'pleasure') 추가
-- 기존 영역 그대로, 신규 영역 1개 + UI 라벨 "사소한 기쁨"
ALTER TYPE goals_area_enum ADD VALUE 'joy';
-- (실제 코드에선 area가 text + CHECK constraint이면 constraint 갱신만)
```

**UI 룰**:
- `plan-write.html`에 영역 옵션 5개로 확장. "사소한 기쁨"은 시각(아이콘) 톤 다운 — 다른 4개와 시각 차이 두지 않음(과시 메커니즘 회피).
- "사소한 기쁨" 영역만은 기간 입력을 **권장이 아닌 옵션**으로. 기본값 "이번 주 / 이번 달" 둘 중 선택.
- `seed.html` 버킷리스트 탭 안에 영역 그리드. 사장님 표현 "코너"는 영역 카드 1개 = "사소한 기쁨" 카드가 별도로 보임.

### 4.3 Axis A·B의 관계 — 사장님이 물은 위계

사장님: "**B를 A의 하위메뉴로 볼 수도 있을 것 같은데... 여튼**."

#### 4.3.1 두 옵션 비교

| 옵션 | 설명 | 평가 |
|---|---|---|
| **R1. A·B를 동등한 두 축** | 라이프 탭 = 일기·버킷리스트·관계 3탭. 풍요로운 삶은 버킷리스트 안의 한 영역. | ○ 모델 정합. "관계"는 새 객체이고 "기쁨"은 기존 객체 확장 — 본질이 다른 두 축. |
| **R2. A(다시 못 깨어난다면) 하위에 B(풍요로운 삶) 두기** | 라이프 탭 = 일기·"내일 다시 못 깨어난다면" 2탭. 후자 안에 (가) 떠오르는 사람 (나) 풍요로운 삶 두 코너. | △ 사장님 발화 톤("하위메뉴로 볼 수도")에 가깝지만, 코드 정합 어색 — 사람(`important_people`)과 기쁨(`goals.area='joy'`)이 서로 다른 모델인데 같은 UI 컨테이너에 들어감. |

**추천 = R1 (동등한 두 축).**

**이유 3줄**:
1. **데이터 모델 차이**: A는 새 1급 객체(`important_people`), B는 기존 `goals` 확장. 한 컨테이너에 묶으면 UI·DB의 결이 어긋남.
2. **사용자 인지 부담 축소**: "관계" 탭 / "버킷리스트" 탭의 라벨이 명확함이 모델의 본질을 그대로 비춤. 하위메뉴는 한 단계 더 깊음.
3. **사장님 발화의 정직한 해석**: "여튼"이라는 말로 사장님 자신도 위계를 미결정으로 두심. **모델 정합이 더 단순한 R1을 추천**하되 사장님 결정으로 최종 확정.

**R1 채택 시 라이프 탭 구조**: 일기 / 버킷리스트 / **관계** (탭 3개). 버킷리스트 안에 "사소한 기쁨" 영역 카드 1개가 추가됨.

## 5. 일기 vs 버킷리스트 — 분리할까

사장님 직관: "**일기를 드러내도 되지 않을지 팀과 상의해봐.**"

### 5.1 사장님이 말한 "드러내라"의 정확한 의미

| 해석 후보 | 의미 |
|---|---|
| (a) 버킷리스트를 일기에서 떼어내 독립 노출하라 | 진입 동선 분리. nav 또는 홈에 버킷리스트 직접 카드. |
| (b) 일기를 더 잘 보이게 하라 | (일기는 이미 첫 탭이므로) → 의미 약함. |
| (c) 일기 안에 묻혀 있는 버킷리스트를 "사소한 기쁨" 등 새 표시로 드러내라 | UI 차원 — 일기 안에 버킷리스트가 시각적으로 융합돼 있어 안 보임. |

코드 사실 확인 후 결론: **사장님 직관은 (a)에 가깝다.** 단, 사장님이 "버킷리스트가 일기에 합해져 있다"고 표현한 부분은 **테이블 통합이 아니라 진입 동선·홈 노출 0의 합성어**.

### 5.2 옵션 비교

| 옵션 | 사용자 인지 부담 | 데이터 모델 변경 비용 | 헌장 톤 | 기존 사용자 영향 |
|---|---|---|---|---|
| **S1. 현행 유지(seed.html 탭 2개)** | 묻힘 그대로 | 0 | 무난 | 0 |
| **S2. 라이프 탭 안 탭 3개(일기/버킷리스트/관계)** | 일목요연. "관계" 새 자리 자연스러움 | `important_people` 신설(one-way) + `goals.area='joy'` 추가(reversible) | 정합 | 적음 (탭 추가만) |
| **S3. 일기·버킷리스트를 각각 독립 페이지로 분리** | 일기·버킷 분리 명확 | 진입 페이지 분리 = nav 변경 또는 홈 카드 추가 부담 | 무난 | 사용자 학습 비용(기존 진입 동선 변화) |
| **S4. 회원 홈에 버킷리스트 카드 추가만 (탭 구조 그대로)** | 묻힘 일부 해소(홈 노출). 깊은 자리는 그대로 | 홈 카드 카탈로그 변경 | 무난 | 0 |

**추천 = S2 + S4 조합.** 라이프 탭에 "관계" 탭 추가(S2) + 회원 홈에 "이번 주 ○○에게" 카드 + "사소한 기쁨" 카드 추가(S4). 일기·버킷리스트의 페이지 분리(S3)는 안 함 — 사용자 학습 비용이 사장님 직관의 이득보다 큼.

### 5.3 정직성

S2의 본질은 "버킷리스트를 드러낸다"가 아니라 **"라이프 탭의 IA를 사장님의 새 축 두 개로 다시 본다"**. 사장님 표현 그대로의 분리(일기/버킷 완전 분리)는 권하지 않는다 — 일기와 버킷리스트는 같은 "내 삶의 기록" 결로 묶이는 게 사용자 인지에 자연스럽기 때문.

### 6.3 권장 메시지 패턴 — 카피라이터 라운드 input

**Axis A nudge 후보**:
- "이번 주 ○○에게 한 번 닿아보시면 어떨까요"
- "마지막 메시지 보낸 게 한 달 전이네요. 짧은 안부 한 줄 어떠세요"
- "○○에게 하고 싶은 말을 적어두셨네요. 오늘 보내보세요"

**Axis B nudge 후보**:
- "이번 주 작은 기쁨 하나, 골라두셨던 것 중에서요"
- "○○이라고 적어두셨네요. 이번 주말은 어떠세요"

**금지 표현 재확인**: `놓치지`, `안 하면`, `벌써`, `미루기만`, `여러분`, `!`(느낌표).

### 6.4 빈도 안전장치

- 푸시 알림 **최대 주 1회**. 2개 이상이면 묶어서 1건.
- 사용자가 1회 무시하면 **다음 주 1회로 자동 연기** (스누즈 로직).
- 3주 연속 무시하면 **자동 일시중지** + 앱 내 안내("푸시를 잠시 쉴게요").
- 헌장 "조용함" 원칙의 운영 룰 직역.

## 7. UX 시나리오 매트릭스

### 7.1 신규 비회원 → 가입 → 첫 행동

- 진입: 인스타 seed-12 카드뉴스 → `beta.html` → `signup.html` → onboarding 7문항.
- onboarding 끝: **A1 옵션 미채택**. 곧장 회원 홈(`index.html`).
- 회원 홈: 오늘 잇고 카드 + "중요한 사람 한 명 적어둘까요?" 카드 (활성 important_people 0건일 때) + 장례희망 카드 + 피드.
- 카드 누르면 → seed.html?tab=people → 첫 한 명 입력 흐름.
- **첫 5분 마찰 평가**: onboarding과 첫 사람 입력을 분리 → 가입 직후 무거움 회피. 사용자가 "준비됐을 때" 자기 발로 클릭.

### 7.2 가입 후 7일째 재방문

- 회원 홈: 오늘 잇고 + "이번 주 ○○에게 한 번 어떠세요" (활성 중 목표 미달 1명) 또는 "사소한 기쁨 하나" (B 영역에 항목 있을 때) + 피드.
- 사용자가 카드 누르면 → seed.html?tab=people 또는 seed.html?tab=goal&area=joy.
- 측정: M1(D30 재방문) + 새 metric **M4. 첫 사람 입력률**(가입 후 D7에 important_people 1건 이상 가진 사용자 비율) — W26 게이트 추가 metric 후보.

### 7.3 사별 페르소나 — Axis A의 슬픔과 행동 넛지의 균형

- 사별 페르소나 신규 가입자: onboarding에서 stage=`bereaved` 진입.
- 첫 회원 홈에서 "중요한 사람 한 명 적어둘까요?" 카드 노출 — **위험**: 떠난 사람이 떠오르면 카드가 자극이 됨.
- **안전장치**:
  - onboarding stage가 `bereaved`이면 첫 30일은 "중요한 사람" 카드 자동 숨김.
  - 사용자가 직접 seed.html?tab=people 진입할 경우, 상단에 "남기고 싶은 분이 있으신가요?" 같은 톤 다운 인사. archived 옵션 우선 안내.
  - `archived` 상태인 사람은 "이번 주 ○○에게" nudge 대상에서 제외.
- **회고의 자리**: 사별한 사람을 `archived` 또는 별도 `legacy` 상태로 두고 "남긴 말" 형태로 기록 가능 (장례희망과 결 일관). 본 라운드 결정 대상은 아님 — 별도 라운드(추모 보류 정책과 연동).

### 7.4 케어링 페르소나 (이미 nest 사용자) — 새 축이 어디서 나타나나

- 케어링 진입은 그대로(nav nest 탭).
- 회원 홈은 다른 사용자와 동일하게 새 카드 노출. **케어링 돌봄 대상(부모)을 important_people에 자동 등록하지 않음** — 케어링은 의무·돌봄 결, 관계는 자발·삶 결로 분리 유지. 별도 객체로 둠.
- 단, 6개월+ 후 사용자 인터뷰에서 "엄마가 important_people이기도 한데 두 곳에 적어야 하나" 같은 신호가 나오면 연동 라운드 별도 필요.

### 7.5 자기성찰 path 사용자 — 새 축과 자기성찰의 관계

- 자기성찰(오늘 잇고)은 답하는 자리, 새 축은 행동하는 자리. 결 분리.
- **시너지 자리**: 오늘 잇고 질문 중에 "지금 가장 보고 싶은 사람은 누구인가요" 같은 질문이 나왔을 때, 답변 후 "이 사람을 관계 목록에 추가할까요?" 한 줄 옵션 노출 가능 — daily_questions 메타데이터에 `link_axis` 컬럼 추가하면 됨. 본 라운드 결정 대상은 아님 (점진 도입).

### 7.6 lifecycle funnel 진입 분기

v3 5장 lifecycle 0(활성·진입) 페르소나의 일상 행동 자리가 **드디어 생긴다** = 그동안 활성 페르소나가 잇다 안에서 할 일이 콘텐츠 정독·오늘 잇고 답변뿐이라 retention이 약했던 구조적 약점이 채워짐. 이게 v3 활성 가설의 진짜 발화점.

## 8. 사장님 결정 카드 (8개)

> 모든 카드는 함께 결정한다. 분리 결정 금지(데이터 모델·UI·노출 룰 상호 의존).

### **D1. Axis A 도입 — 떠오르는 사람 + 연락 빈도 + 행동 넛지**
- **결정**: 도입 / 미도입 / Axis B만 도입
- **추천**: **도입.** 사장님 통찰의 가장 핵심.
- **영향**: `important_people` 테이블 신설 = **one-way door**. 백업 + 창업자 승인 + PE 라운드.
- **반대 견해**: 행동 격차(intention-action gap)로 인해 "적어두기만 하고 안 보내는" 사용자 비율이 높을 수 있음. 추가 측정 metric(M4 첫 입력률, M5 last_contact_at 갱신율)으로 검증.

### **D2. Axis A·B 위계 — 동등 두 축 vs A 하위 B**
- **결정**: **R1 동등** / R2 하위
- **추천**: **R1 (동등 두 축).**
- **영향**: 라이프 탭 = 일기·버킷리스트·관계 3탭. 사장님 발화 결의 정확 매핑이 R2지만, 모델·UI 정합은 R1이 우위.

### **D3. Axis B 모델 — 기존 `goals` 영역 확장 vs 별도 테이블**
- **결정**: B1(영역 추가) / B2(별도 테이블) / **B3(영역 추가 + UI 분리)**
- **추천**: **B3.**
- **영향**: `goals.area` enum에 `joy` 추가 (reversible). plan-write.html 영역 옵션 1개 추가. 사용자 인지상 "사소한 기쁨" 코너로 노출.

### **D4. 라이프 탭 IA 재설계 — 탭 2개에서 3개로**
- **결정**: 현행(일기·버킷리스트) / **신규(일기·버킷리스트·관계)** / 다른 구조
- **추천**: **신규 3탭.**
- **영향**: `seed.html` 탭 segment 1개 추가. seed.html 진입 시 기본 탭은 일기 그대로(기존 사용자 동선 보호).

### **D5. 회원 홈 카드 카탈로그 — Axis A·B 카드 추가**
- **결정**: 추가 / 미추가 / 일부만(A만 또는 B만)
- **추천**: **둘 다 추가**. 우선순위 알고리즘은 (1) Axis A 목표 미달 1명 (2) Axis B 사소한 기쁨 1개 (3) 둘 다 없으면 "한 명 적어둘까요?" 입구 카드.
- **영향**: index.html 회원 홈 카드 자리 변경. 기존 카드(오늘 잇고·장례희망)와 새 카드의 시각 위계 결정 필요 — 카피라이터·PE 라운드.

### **D6. 행동 넛지 채널 — in-app only vs in-app + 푸시 옵트인**
- **결정**: in-app only / **in-app + 푸시 옵트인 기본 OFF**
- **추천**: **in-app + 푸시 옵트인 OFF 기본**. 푸시는 주 1회 상한 + 3주 연속 무시 시 자동 중지.
- **영향**: 기존 push-notify 인프라 재사용. 옵트인 UI(root.html 또는 seed.html 사람 코너 안)에 토글 1개 추가.

### **D7. 사별 페르소나 안전장치**
- **결정**: 도입 / 미도입
- **추천**: **도입**. onboarding stage `bereaved` 사용자는 가입 후 30일간 "사람 적어둘까요?" 카드 자동 숨김 + archived 안내 우선.
- **영향**: index.html 카드 렌더 분기 1줄 + seed.html 사람 탭 상단 인사 분기. 헌장 톤 정합.

### **D8. 새 metric 등재 (W26 게이트)**
- **결정**: 추가 / 미추가
- **추천**: **추가**. 기존 M1·M2·M3에 **M4 첫 사람 입력률(가입 D7에 important_people 1건 이상)**, **M5 last_contact_at 갱신율(활성 중 주 1회 이상 토글 비율)** 등재.
- **영향**: 이벤트 로그 설계에 같이 동봉(2026-06-11 D5·D7 라운드).
