# 앱 기획·개발 원칙 (창업자 지정)

> 2026-07-06 창업자 확정. **모든 앱(잇다·묻다·이후 신규 앱) 기획·개발 세션은
> 시작 전에 이 문서를 따른다.** 위반이 필요한 경우 창업자 승인을 먼저 받는다.

## 원칙

1. **배포 스크립트는 공통 재사용** — 앱마다 배포 스크립트를 새로 만들지 않는다.
   현재 공통 경로: `.github/workflows/pages.yml` (main → gh-pages 동기화,
   서버 전용·내부 폴더 자동 제외). 새 앱은 이 파이프라인에 올라탄다.

2. **새 앱은 기존 디자인 시스템부터 확인** — 화면을 그리기 전에
   `docs/company/design-system.md` 와 토큰(`auth-styles.css` `:root`)을 읽는다.
   토큰만 사용, 임의 hex 금지. (묻다 사례: `mutda/css/mutda.css`가 잇다 토큰을
   그대로 상속 — 신규 앱도 같은 방식.)

3. **개인정보처리방침은 공통 템플릿 사용** — 앱마다 방침 문서를 따로 쓰지 않는다.
   공통 템플릿을 한 곳에 두고 각 앱은 링크만 연결한다.
   ※ 현재 상태: 잇다는 nav.js 푸터에 개인정보 링크 스텁만 존재. 공통 템플릿
   문서(`docs/company/privacy-template.md` 또는 공용 페이지) 제작이 선행 과제 —
   신규 앱에서 방침이 필요해지는 시점에 먼저 만든다.

4. **앱 간 중복 코드 금지** — 같은 기능(인증, 내비, 토스트, 푸시, 공유 등)을
   앱마다 복붙하지 않는다. 두 번째로 필요해지는 순간 공통 모듈로 추출한다.
   ※ 현재 알려진 중복(상환 대상): 잇다 `auth.js`/`js/pwa.js` ↔ 묻다
   `mutda/js/app.js`/`mutda/js/push.js` — 구조가 유사하다. 다음 신규 앱 착수 전에
   공통 패키지(`shared/`)로 추출을 검토한다.

## 함께 지켜지는 상위 규칙 (기존 문서)

- 헌장: `docs/company/CHARTER.md` — 조용함·존엄, 과시·경쟁 금지, YMYL 전문가 감수.
- 세션 규칙: `CLAUDE.md` — 워크로그, 배포 순서, 캐시 버전, 되돌리기 어려운 행동 승인.
- 디자인: `docs/company/design-system.md`, 프로덕트 원칙:
  `docs/company/product-design-principles.md`.
