# 결정 0001 — 콘텐츠를 별도 어드민으로 관리할 것인가

- 일자: 2026-05-25
- 결정자: 전략가(오케스트레이터/리더)
- 문 유형: **2-way door** (되돌릴 수 있음 — 문서·프런트 범위)
- 상태: 결정 완료, 실행

## 질문
콘텐츠를 앱에 직접 쓰지 않고 별도 admin 페이지에 넣고 수정해야 지속가능한 관리가 되지 않을까? 지금 단계에서 어드민 페이지 별도 구축이 필요한가?

## 팀 관점
- **전략가**: 개념적으로는 맞다(CMS 분리는 정석). 그러나 지금은 검증 단계다 — 콘텐츠 ~5편, 에디터 1명(창업자), 6/1 비공개 20인 테스트. "만들기 전에 묻는다 / 검증 우선" 원칙상, 아직 안 아픈 곳에 시스템을 짓는 건 조기 최적화다.
- **PE**: 핵심 사실 — 별도 admin은 **이미 불필요하다.** `content-write.html`이 `?id=`로 **수정·삭제를 이미 지원**한다(create/edit/delete 완비). 진짜 "지속가능성"의 핵심인 버전관리·리뷰·출처/YMYL 보존은 **repo의 `docs/content/*.md`가 단일 원본(source of truth)** 역할을 하며 git이 그 일을 한다. 남은 갭은 단 하나: `createContent`가 `author_type:'user'`로 하드코딩되어 '공식' 배지를 정식으로 못 단다. 지금은 계정명을 "잇다 에디터"로 바꿔 우회 중.
- **운영**: 편당 발행 빈도가 낮은 지금, 워크플로(저장→리뷰→발행)에 사람 손이 더 안 든다. 어드민 구축 비용 > 절감 효과.
- **마케팅**: 독자가 보는 건 렌더 품질·읽기 UX·바이라인이지 입력 도구가 아니다. 그쪽은 이미 끝냄.

## 결정
1. **별도 어드민 콘텐츠 페이지는 지금 짓지 않는다.** 기존 `content-write.html`(create/edit/delete)로 충분.
2. **`docs/content/*.md` = 콘텐츠 단일 원본.** 앱은 발행/렌더 대상일 뿐. 수정은 repo 원본을 먼저 고치고 앱에 반영(또는 그 반대 시 repo도 동기화)해 드리프트 방지.
3. **author_type 갭은 우회(에디터 계정명)로 유지.** '공식' 정식 배지 경로는 DB/RPC 변경 = **1-way door** → 검증 이후, 창업자 승인 시에만.

## 어드민 구축 트리거 (이때 재검토)
아래 중 하나라도 충족되면 어드민 CMS를 Phase-1 과제로 승격한다.
- 에디터가 **2명 이상**(비개발자 포함)
- 발행 콘텐츠 **20편 초과** 또는 주 2편 이상 정기 발행
- 카테고리/태그·예약발행·승인 워크플로 등 **편집 거버넌스**가 필요해질 때

## 실행
- 본 결정 기록 커밋.
- 추가 빌드 없음(2-way door 범위 내). DB·RPC 변경 없음.

## 📌 미래 트리거 — 설계기록(ADR 상시화) 도입 재검토

> 기록 일자: 2026-06-05 · 출처: 창업자와의 대화(설계기록 포맷 도입 논의)

**맥락.** 오늘의집 류의 Google-style 설계문서(Background/Objective/Design Ideas + 날짜별
변경 로그)를 잇다 코딩 전·중·후에 상시 작성할지 검토했다. **결론: 지금은 하지 않는다.**
잇다는 정적 HTML + 바닐라 JS + Supabase, 빌드 없음, 실질 단일 결정자 — 그 무게의 문서
프로세스는 헌장의 *"안 아픈 곳에 시스템 짓지 않기 / 검증 우선"* 에 어긋난다. 또한 그 포맷의
핵심(왜·결정·door 유형·재검토 트리거·날짜 로그)은 **이미 이 `docs/decisions/` 패턴과
`docs/worklog.md` 가 나눠 수행**하고 있다.

**아래 중 하나라도 충족되면, 세션이 ADR 상시화(쓰는 기준 명문화 + 경량 템플릿 +
`CLAUDE.md`에 "1-way door 변경은 ADR 먼저" 규칙 추가)를 창업자에게 제안한다.**
- 코드 기여자가 창업자 외 **2명 이상**
- 동시 진행 기능 **3개 이상**, 또는 세션 간 충돌·되돌리기 사고가 **실제로 발생**
- Supabase 스키마/RPC 변경을 **되돌리느라 시간을 날린 사건**이 한 번이라도 발생
- 외부에 API/연동을 **여러 팀이 공유**하기 시작

충족 전에는 손대지 않는다. 이 블록은 약속을 세션 간에 보존하기 위한 표식일 뿐이다.
